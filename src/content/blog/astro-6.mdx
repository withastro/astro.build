Astro 6 is here! 🎊

Following a two-month beta, Astro 6 introduces a broad set of new capabilities across developer experience, deployment, performance, and security.

Highlights include a redesigned `astro dev`, Astro on Cloudflare Workers, a built-in Fonts API, Live Collections, and stable Content Security Policy.

Alongside these features, we completed a major refactor of Astro's dev server and much of the build command to bring dev and production closer together. The result is fewer "works in dev, breaks in prod" surprises today, plus a platform that lets us ship ambitious improvements faster throughout upcoming Astro 6.x releases.

Here are the highlights:

One more thing: Astro 6 includes an experimental new **Rust compiler** for `.astro` components. It's an early step, but an important one. As we continue investing in Rust-powered parts of Astro, we're unlocking better performance, stronger scalability for large sites, and a faster path to bigger improvements in upcoming 6.x releases.

To upgrade an existing project to Astro 6, use the automated `@astrojs/upgrade` CLI tool:

```sh
# Recommended:
npx @astrojs/upgrade

# Manual:
npm install astro@latest
```

New projects, simply

```sh
npm create astro@latest
```

## A redesigned `astro dev`

We've redesigned `astro dev` to better align with production.

- Vite Environment API refactor, closing the gap between dev and prod
- Benefits: run against real runtimes, support more platforms, unified code paths
- Dev and prod now share the same code paths, eliminating subtle env-only bugs -- We got several during this refactor

### Astro on Cloudflare Workers

Cloudflare users now get workerd in development, prerendering, and in production.

- First-class workerd support in dev via Vite Environment API
- Real platform APIs: Durable Objects, KV, R2, Workers Analytics Engine
- Access bindings via `cloudflare:workers` module (no more `Astro.locals.runtime`)
- `astro preview` now works with Cloudflare
- Cloudflare Pages dropped in favor of Workers

## Built-in Fonts API

Almost every website these days uses custom fonts. However, it can be surprisingly complicated to get them right: Why do I get a Flash of Unstyled Text? Should I add a preload link? What about preconnect links? Do I self-host or rely on a third-party provider?

Astro provides a way to use fonts from your filesystem and various font providers (e.g. Fontsource, Google) through a unified, fully customizable, and type-safe API.

Web fonts can impact page performance at both load time and rendering time. This API helps you keep your site performant with automatic web font optimizations including preload links, optimized fallbacks, and opinionated defaults.

The Fonts API focuses on performance and privacy by downloading and caching fonts so they’re served from your site. This can avoid sending user data to third-party sites, and also ensures that a consistent set of fonts is available to all your visitors.

To enable this feature, configure an `fonts` object with one or more fonts:

```js title="astro.config.mjs"
import { defineConfig, fontProviders } from 'astro/config';

export default defineConfig({
	fonts: [
		{
			provider: fontProviders.fontsource(),
			name: 'Roboto',
			cssVariable: '--font-roboto',
		},
	],
});
```

Then, add a `<Font />` component and site-wide styling in your `<head>`:

```astro title="src/components/Head.astro"
---
import { Font } from 'astro:assets';
---

<Font cssVariable="--font-roboto" preload />
<style>
	body {
		font-family: var(--font-roboto);
	}
</style>
```

Head over to the [fonts guide](https://docs.astro.build/en/guides/fonts/) to learn more.

## Live Collections

Use the Content Layer API you love with live collections that don't require a rebuild.

- Configuration with defineLiveCollection in src/live.config.ts
- Query with getLiveCollection() and getLiveEntry(), with built-in error handling
- Supports filters and custom queries if loader supports them

## Content Security Policy

Content Security Policy is now stable, and we're proud to say that Astro is the first meta-framework to support it across multiple environments and platform. The Astro way.

Enabling CSP in Astro is easy as every feature, and you can opt-in using Astro's sensible defaults. The `csp` flag has been moved under the `security` setting:

```js ins={4-6} del={7-9}
import { defineConfig } from 'astro/config';

export default defineConfig({
	security: {
		csp: true,
	},
	experimental: {
		csp: true,
	},
});
```

By setting the option to `true`, many of the Astro features are supported out of the box. Additional options can use used to customize your CSP experience. You can change the hashing algorithm, add more directives and hashes, if you have external scripts or styles:

```js
import { defineConfig } from 'astro/config';

export default defineConfig({
	security: {
		csp: {
			algorithm: 'SHA-512',
			directives: [
				"default-src 'self'",
				"img-src 'self' https://images.cdn.example.com",
			],
			styleDirective: {
				hashes: ['sha384-styleHash'],
			},
			scriptDirective: {
				hashes: ['sha384-scriptHash'],
			},
		},
	},
});
```

Astro implements CPS by hashing all script and styles shipped in page; this technique allows Astro support static pages and dynamic pages. Astro injects the CSP header using `<meta http-equiv="Content-Security-Policy">` in static pages, while dynamic pages are served using a `Request` header. If your adapter supports the adapter feature `staticHeaders`, CSP is served via `Request` header for static pages too.

As part of the stabilization, CSP is now supported out of box for [responsive images](https://docs.astro.build/en/reference/configuration-reference/#imageresponsivestyles). The styles for images are calculated at build-time, and the styles are applied using a mix CSS classes and `data-*` attributes.

## Experimental: Queued Rendering

Queued rendering leverage a new rendering strategy that is based on a queue. This new strategy is designed to replace the current strategy in Astro v7.

When Astro tries to render a JSX element or an Astro component, it does so with a recursion. The rendering functions are called inside themselves.

While recursion works most of the time, sometimes it can create some memory pressure, especially when working with tree-like structures.

With queued rendering, the components of the tree are first collected in an ordered list, and then they are iterated and rendered. Some early benchmarks have shown improvements up to 2x in rendering speed.

Enable the new strategy by enabling the experimental flag:

```js ins={5-7}
import { defineConfig } from 'astro/config';

export default defineConfig({
	experimental: {
		queuedRendering: {
			enabled: true,
		},
	},
});
```

Queued rendering comes with two other sub-features: node pooling and content cache.

### Node Pooling

This is a technique to save node types inside an internal cache while rendering multiple pages. The cache is populated during the rendering of a page, and reused when the next page is rendered.

This feature is exclusively meant when building an Astro website, so by design this feature is automatically turned off when rendering dynamic pages.

Node pooling is automatically enabled when you enabled queued rendering. If you wish to disable it, you can set the size of the pool to zero. Or you can increase its default value.

```js ins={7} ins="3000"
import { defineConfig } from 'astro/config';

export default defineConfig({
	experimental: {
		queuedRendering: {
			enabled: true,
			poolSize: 3000,
		},
	},
});
```

### Content cache

A feature that allows caching the content generated by components. This is particularly useful when rendering small components that return the same content. Their value is cached so that it doesn't need to be recomputed again.

The feature is turned off by default, and you can enable it by setting `contentCache` to `true`:

```js ins={7}
import { defineConfig } from 'astro/config';

export default defineConfig({
	experimental: {
		queuedRendering: {
			enabled: true,
			contentCache: true,
		},
	},
});
```
