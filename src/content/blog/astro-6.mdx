**Astro 6 is here!** Astro 6 introduces a broad set of new capabilities including a built-in Fonts API, Content Security Policy API, and support for Live Content Collections that work with your externally-hosted content through the unified Astro content layer.

Alongside these new features, we also completed a major refactor of the Astro dev server and much of the build pipeline. Powered by Vite's new Environment API, Astro can now run your exact production runtime during development. That means fewer "works in dev, breaks in prod" surprises — especially on non-Node.js runtimes like Cloudflare Workers, Bun, and Deno.

**One more thing**: Astro 6 includes an experimental new Rust compiler — the successor to our original Go-based `.astro` compiler. It's still early, but the results are already impressive (and in some cases, even more reliable than our current Go compiler). We'll continue to invest in Rust-powered tooling throughout the 6.x release line to improve performance and scalability for large sites.

To upgrade an existing project to Astro 6, use the automated `@astrojs/upgrade` CLI tool:

```sh
# Recommended:
npx @astrojs/upgrade

# Manual:
npm install astro@latest
```

For new projects, simply use:

```sh
npm create astro@latest
```

## A redesigned `astro dev`

Astro 6 includes a major refactor of the development server to bring development and production much closer together.

As Astro evolved from static-first toward broader server rendering support, dev and prod ended up following different internal paths. Over time, that divergence created edge-case behavior that was difficult to reproduce and debug across environments.

In Astro 6, we reworked `astro dev` around Vite's Environment API so Astro can run against real runtimes during development, with shared code paths that better reflect production behavior. The result is a more predictable development experience and fewer bugs caused by environment-specific differences.

### Astro on Cloudflare Workers

The clearest example of this shift is the rebuilt `@astrojs/cloudflare` adapter.

With Astro 6, Cloudflare's `workerd` runtime now powers development, prerendering, and production. Instead of relying on the old `Astro.locals.runtime` simulation APIs, you can develop directly against Cloudflare's platform APIs using `cloudflare:workers`, including environment bindings, Durable Objects, KV, and R2.

That means behavior in development is far closer to what runs in production, so issues are caught earlier and platform-specific features are easier to build with confidence.

## Built-in Fonts API

Almost every website these days uses custom fonts. However, it can be surprisingly complicated to get them right: Why do I get a Flash of Unstyled Text? Should I add a preload link? What about preconnect links? Do I self-host or rely on a third-party provider?

Astro provides a way to use fonts from your filesystem and various font providers (e.g. Fontsource, Google) through a unified, fully customizable, and type-safe API.

Web fonts can impact page performance at both load time and rendering time. This API helps you keep your site performant with automatic web font optimizations including preload links, optimized fallbacks, and opinionated defaults.

The Fonts API focuses on performance and privacy by downloading and caching fonts so they’re served from your site. This can avoid sending user data to third-party sites, and also ensures that a consistent set of fonts is available to all your visitors.

To enable this feature, configure an `fonts` object with one or more fonts:

```js title="astro.config.mjs"
import { defineConfig, fontProviders } from 'astro/config';

export default defineConfig({
	fonts: [
		{
			provider: fontProviders.fontsource(),
			name: 'Roboto',
			cssVariable: '--font-roboto',
		},
	],
});
```

Then, add a `<Font />` component and site-wide styling in your `<head>`:

```astro title="src/components/Head.astro"
---
import { Font } from 'astro:assets';
---

<Font cssVariable="--font-roboto" preload />
<style is:global>
	body {
		font-family: var(--font-roboto);
	}
</style>
```

Head over to the [fonts guide](https://docs.astro.build/en/guides/fonts/) to learn more.

## Live Collections

Live Collections are now stable in Astro 6, bringing Content Collections to content that changes frequently, without requiring a rebuild.

Content Collections (introduced in Astro 2.0) gave Astro a type-safe way to work with local content, and Astro 5.0 expanded that model with loaders for remote sources. Live Collections extend that evolution by letting you fetch frequently updated content at request time with the same collection-first workflow.

Use `defineLiveCollection()` to define a live source in `src/live.config.ts`:

```ts title="src/live.config.ts"
import { defineLiveCollection } from 'astro:content';
import { z } from 'astro/zod';
import { cmsLoader } from './loaders/my-cms';

const updates = defineLiveCollection({
	loader: cmsLoader({ apiKey: process.env.MY_API_KEY }),
	schema: z.object({
		slug: z.string(),
		title: z.string(),
		excerpt: z.string(),
		publishedAt: z.coerce.date(),
	}),
});

export const collections = { updates };
```

Then query live content in your page with built-in error handling:

```astro
---
import { getLiveEntry } from 'astro:content';

const { entry: update, error } = await getLiveEntry(
	'updates',
	Astro.params.slug,
);

if (error || !update) {
	return Astro.redirect('/404');
}
---

<h1>{update.data.title}</h1>
<p>{update.data.excerpt}</p>
<time>{update.data.publishedAt.toDateString()}</time>
```

That means editorial content like announcements, changelogs, and documentation updates can appear immediately, while still benefiting from Astro's type-safe content APIs.

## Content Security Policy

Content Security Policy is now stable, and we're proud to say that Astro is the first meta-framework to support it for both static and dynamic pages, as well as in both server and serverless environments... in classic Astro fashion.

Enabling CSP in Astro is easy as every feature, and you can opt-in using Astro's sensible defaults. The `csp` flag has been moved under the `security` setting:

```js ins={4-6} del={7-9}
import { defineConfig } from 'astro/config';

export default defineConfig({
	security: {
		csp: true,
	},
	experimental: {
		csp: true,
	},
});
```

By setting the option to `true`, many of the Astro features are supported out of the box. Additional options can use used to customize your CSP experience. You can change the hashing algorithm, add more directives and hashes, if you have external scripts or styles:

```js
import { defineConfig } from 'astro/config';

export default defineConfig({
	security: {
		csp: {
			algorithm: 'SHA-512',
			directives: [
				"default-src 'self'",
				"img-src 'self' https://images.cdn.example.com",
			],
			styleDirective: {
				hashes: ['sha384-styleHash'],
			},
			scriptDirective: {
				hashes: ['sha384-scriptHash'],
			},
		},
	},
});
```

Astro implements CSP by hashing all script and styles shipped in page; this technique allows Astro support static pages and dynamic pages. Astro injects the CSP header using `<meta http-equiv="Content-Security-Policy">` in static pages, while dynamic pages are served using a `Request` header. If your adapter supports the adapter feature `staticHeaders`, CSP is served via `Request` header for static pages too.

As part of the stabilization, CSP is now supported out of box for [responsive images](https://docs.astro.build/en/reference/configuration-reference/#imageresponsivestyles). The styles for images are calculated at build-time, and the styles are applied using a mix CSS classes and `data-*` attributes.

## Upgraded Packages

The following are major upgrades for dependencies used by all Astro projects:

### Vite 7

Astro 6 and packages under the `@astrojs` npm organization now use Vite 7.

If your project pins a custom Vite version, update it to v7 or later before upgrading. We hope to have support for the upcoming Vite 8 in a minor version of Astro 6.x.

### Shiki 4

Astro now uses Shiki 4 for code highlighting in the `<Code />` component and for Markdown/MDX code blocks.

### Zod 4

Astro 6 upgrades content schema validation to Zod 4.

When defining schemas, import Zod from `astro/zod` rather than `astro:content`.

See the [upgrade guide](https://docs.astro.build/en/guides/upgrade-to/v6/#zod-4) for tips on common things that need to change with Zod 4.

## Experimental: Rust Compiler

Astro 6 introduces an experimental new Rust compiler for `.astro` components — and it's fast.

This is one of the biggest bets in this release: faster compilation, stronger diagnostics, and a path to scaling Astro further for larger and more complex sites.

What started as an AI experiment while updating our Go-based compiler for Astro 6 quickly moved from "can this work?" to "this is actually good enough to ship.". We hope to make it the default compiler in a future major release of Astro.

You can try it today as an experimental feature by enabling the `rustCompiler` flag in your Astro config, and installing the `@astrojs/compiler-rs` package in your project:

```sh
npm install @astrojs/compiler-rs
```

```js
import { defineConfig } from 'astro/config';

export default defineConfig({
	experimental: {
		rustCompiler: true,
	},
});
```

We're actively exploring more Rust-powered parts of Astro in order to improve performance, with more to share soon.

Read our reference on [the Rust compiler](https://docs.astro.build/en/reference/experimental-flags/rust-compiler/) for more details.

## Experimental: Queued Rendering

Astro 6 introduces an experimental new rendering strategy with some early benchmarks showing improvements of up to 2x in rendering speed!

Queued rendering is based on a two-pass process where the first pass traverses the tree of components, emits an ordered queue, and then the queue is rendered. This has led to both speed and memory improvements over the current recursion-based process, where rendering functions end up being called inside themselves.

This new strategy is designed to replace the current strategy in Astro v7.

Enable the new strategy by enabling the experimental flag:

```js ins={5-7}
import { defineConfig } from 'astro/config';

export default defineConfig({
	experimental: {
		queuedRendering: {
			enabled: true,
		},
	},
});
```

This new rendering engine can be further customized with additional sub-features: node pooling and content cache. See the [experimental queued rendering documentation](https://docs.astro.build/en/reference/experimental-flags/queued-rendering/) for more on this feature.

## Experimental: Route Caching

Astro 6 includes an experimental route caching API that gives you a platform-agnostic way to cache server-rendered responses using web standard cache semantics.

Today, if you want to cache SSR responses, you either set headers manually or rely on adapter-level conventions such as ISR, that vary between hosts. Route caching gives you a single API that works the same way everywhere: you set caching directives in your routes, and Astro translates them into the appropriate headers or runtime behavior depending on your adapter.

Enable route caching by configuring a cache provider in your Astro config:

```js
import { defineConfig } from 'astro/config';
import { memoryCache } from 'astro/config';

export default defineConfig({
	experimental: {
		cache: {
			provider: memoryCache(),
		},
	},
});
```

Then use `Astro.cache` in `.astro` pages or `context.cache` in API routes to control caching per-request:

```astro
---
Astro.cache.set({
	maxAge: 120, // Cache for 2 minutes
	swr: 60, // Serve stale for 1 minute while revalidating
	tags: ['home'], // Tag for targeted invalidation
});
---

<html><body>Cached page</body></html>
```

You can also invalidate cached entries by tag or path:

```ts
await context.cache.invalidate({ tags: ['data'] });
await context.cache.invalidate({ path: '/api/data' });
```

Route caching works alongside [live collections](#live-collections) to allow automatic tracing of dependencies between pages and content entries. When a content entry is updated, any cached responses that depend on it can be automatically invalidated. To associate a page with a content entry, use `cache.set(entry)` in your page:

```astro
---
import { getEntry } from 'astro:content';
const product = await getEntry('products', Astro.params.slug);
Astro.cache.set(product);
---

<html><body>{product.data.name}</body></html>
```

To invalidate all cached responses that depend on a content entry, use `cache.invalidate(entry)`:

```ts
import { getLiveEntry } from 'astro:content';

export async function POST({ cache }) {
	const { entry } = await getLiveEntry('products', 'featured');
	if (entry) {
		await cache.invalidate(entry);
	}
	return Response.json({ ok: true });
}
```

To set default cache options without needing to add code to every page, you can set `experimental.routeRules` that apply caching rules based on route patterns:

```js
import { defineConfig } from 'astro/config';
import { memoryCache } from 'astro/config';

export default defineConfig({
	experimental: {
		cache: {
			provider: memoryCache(),
		},
		routeRules: {
			'/api/*': { swr: 600 },
			'/products/*': { maxAge: 3600, tags: ['products'] },
		},
	},
});
```

This initial release includes a simple in-memory cache provider, which works best with the Node.js adapter. We will be adding cache providers for all of Astro's supported deployment platforms in the next few weeks. We also hope there will be community-built providers for CDN services, and other storage types such as Redis. Get in touch [on Discord](https://astro.build/chat) if you're interested in building one!

For more details, see the experimental [route caching docs](https://docs.astro.build/en/reference/experimental-flags/route-caching/).
